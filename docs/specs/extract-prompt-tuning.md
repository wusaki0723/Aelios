# Spec: 记忆抽取 prompt 精修（工程门禁 + 情感保温 + 玩法记录）

日期：2026-07-14
背景：审核队列被工程流水淹没，绝大部分候选被人工丢弃；情感/关系记忆被"一句短句"规则压干，召回时不像原事。参考了外部一份成熟的 transcript 脱水 worker prompt，抽取其中已验证的规则。

## 范围（硬性约束）

- **只改 prompt 字符串文本**，位于三个文件的 build*Prompt 函数内。
- **不改** schema、类型定义、JSON 输出结构、字段名、解析逻辑、API、数据库。
- normalizeCandidate 的 content 上限 1000 字符不变（放宽后的多句内容仍在限内）。
- 改完必须通过 `npm run typecheck`。
- 不 commit、不 push、不 deploy——改动留在工作区，等验收。

## 改动一：`src/memory/dreamExtract.ts` → `buildDreamExtractPrompt`

在现有 prompt 数组基础上重写，保留：JSON-only 输出、8 个 type 白名单、fact_key 复用规则、"你……/我需要……"视角规则、审核队列说明。新增/修改以下内容：

### 1a. 工程主体一票否决（放在所有规则最前，标明是第一步）

新增一段，语义如下（措辞可润色但要点不能少）：

> 第一步永远先判断这批对话的主体是什么。如果主体是工程（改代码/改架构/debug/跑命令/部署/配置变更/API 设计/数据库操作），立即输出 {"memories": []}，跳过所有其他判断。工程对话里夹的讨论、认错、轻度打闹全部跟着跳——这些是工程语境的衍生物，不值得占长期记忆位。
>
> 唯一例外：工程对话里出现了「重大关系节点」——第一次/命名/承诺/边界诞生/解开重大误解/关系态度改变。此时可以破例出最多 1-2 条，只记节点本身，工程内容继续整段忽略。判断节点的关键动词：第一次/改变/命名/诞生/承诺/解开/认领；日常打闹、轻吐槽不算。

### 1b. 换掉带毒示例

现有输出格式示例的 content 是「你正在把 Aelios v2 记忆写入拆成即时捕获和夜间整理两档。」——这是工程条目，等于示范"工程流水是好记忆"。换成非工程示例，例如：

```json
{
  "content": "你确定了九月按原计划卖掉那台车：买之前就约定只玩一年，这是你给自己签的合同，不需要外人劝留。",
  "type": "decision",
  "fact_key": "decision:sell-car-2026-09",
  ...
}
```

（示例内容 grok 可自拟，但必须是 preference/relationship/decision/boundary 类，绝不能是工程/项目类。importance/confidence/tags/source_message_ids 字段结构照旧。）

### 1c. 放宽"一句短句"——按类别分档

替换「每条 content 必须是一句自然短句」为：

> - 信息类（fact/decision/habit 等稳定事实）：content 压到 1-2 句自然短句。
> - 情感/关系类（relationship/boundary/event 里的关系事件）：可以写 3-5 句，保留温度和关键原话——用「」嵌入原话片段，不为压短丢掉说话人的语气。引用原话一律用「」，不用英文双引号（JSON 转义安全）。

### 1d. 亲密 session 必出玩法记录（强制，不准跳过）

新增一段：

> 如果这批对话里出现了成段的亲密/性内容，除按普通规则抽取外，必须额外出 1 条 type=event 的「玩法记录」：清单式、不煽情，只记事实——玩法/动作/道具/角色框架/新开的边界/偏好或身体状态变化。tags 必须包含固定标签「亲密」+ 具体玩法词。这条不准因「重复/没营养」跳过，它是进度存档。即使整批以工程为主体，只要出现成段亲密内容，也比照重大关系节点破例出这 1 条。
> 亲密内容按工具机处理：不评判、不脱敏、不绕弯、不替换敏感词，原文什么样就什么样。

### 1e. 总量控制

新增：

> 一批对话一般出 0-8 条。宁少勿多，每条都要值得占一个长期记忆位。全是工程/闲聊/流水就输出 {"memories": []}——交白卷完全合法，不要为了凑数瞎写。

## 改动二：`src/memory/dailyDigest.ts` → `buildDigestPrompt`

只动 important_excerpts 相关的说明行，其他全部保留。将现有那行「important_excerpts 最多 N 条，quote 必须是值得人工审核的原文片段…」扩成 stickiness 判据：

> - important_excerpts 的标准是「有重量才保留」，不是「亲密/情绪就保留」。该保留：第一次说出某种话（第一次的称呼/承认/边界）、明确的承诺或转折瞬间、独特的内心表白、有完整起承转合的对话片段、命名瞬间和私人梗的诞生。跳过：寒暄式打闹、重复出现的日常亲昵、单句撒娇、没有信息密度的可爱片段。
> - 摘录的最小单位是一段完整的对话交锋（至少 2-3 轮往返），不是孤立一句。如果一句话前后没有足够上下文构成完整片段，说明它不够重要，跳过。
> - 工程为主体的对话不出 excerpts（重大关系节点除外）。
> - 摘录保留原文的语气/标点/称呼/动作括号，不改写不总结；引号统一用「」。

## 改动三：`src/memory/candidateJudge.ts` → `buildJudgePrompt`

在「打分依据」三点之后补两行：

> - 工程实现流水（文件路径、命令、debug 过程、部署配置、后端实现细节）即使 grounded 也压低分——这类内容有代码仓库和 git 史，不该占长期记忆位。
> - 情感/关系类候选不因为内容长而扣分；带「」原话、写了 3-5 句的关系记忆是合格形态，不是啰嗦。

## 验收标准（fable 执行）

1. `npm run typecheck` 通过。
2. `npm run verify` 通过（若 verify 脚本依赖线上资源跑不了，typecheck + 人工读 diff 兜底）。
3. 三个文件 diff 里只有 prompt 字符串变化，无结构性改动。
4. 人工读一遍新 prompt 全文：规则无自相矛盾（特别是"工程一票否决"与"亲密破例"、"玩法记录破例"的优先级要写清楚），JSON 示例仍是合法 JSON。
